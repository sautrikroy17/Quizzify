Encapsulation

Hiding data inside a class — accessible only through a safe, controlled interface
📖 Definition

Encapsulation is the principle of bundling data (variables) and the methods that operate on that data into a single unit (class), and restricting direct access to some of the object's components. Only selected methods are exposed as a public interface.

QuizzifyCore.cpp — Lines 23–39ENCAPSULATION
class Question {
private:                       // 🔒 LOCKED from outside
    string text;
    vector<string> options;
    int correctOptionIndex;   // ← answer is HIDDEN

public:                        // ✅ Safe interface
    // Constructor
    Question(string q, vector<string> opts, int idx)
        : text(q), options(opts),
          correctOptionIndex(idx) {}

    string getText() const { return text; }
    vector<string> getOptions() const { return options; }

    bool checkAnswer(int selectedIndex) const {
        return selectedIndex == correctOptionIndex;
    }                          // ← You can CHECK but never READ
};
✅ Why Used Here

The correctOptionIndex stores the answer to a quiz question. If it were public, any code could read the answer directly — defeating the purpose of a quiz. By making it private, the only way to interact with the answer is through checkAnswer(), which returns only true/false.

💡 Real-World Analogy

Like an ATM — it has your bank balance stored internally (private), but you can only check it through the screen interface (public method). You cannot reach inside and modify the number directly.

❌ Without Encapsulation

Someone could write q.correctOptionIndex = 0; and force every answer to be option A — the quiz is broken.

MemberAccess LevelReason
textprivateQuestion text shouldn't be altered after creation
optionsprivateOptions list is fixed — immutable after construction
correctOptionIndexprivateThe answer must be hidden to prevent cheating
getText()publicUI needs to display the question text — read-only access
checkAnswer()publicValidates a student's answer without revealing the correct index

Abstraction

Defining WHAT must exist, without specifying HOW it works
📖 Definition

Abstraction means exposing only the essential features of an object and hiding the complex implementation details. In C++, this is achieved through pure virtual functions (making abstract classes) and interfaces (classes with only pure virtual functions).

Two layers of abstraction in this file:

Lines 13–17 — InterfaceLAYER 1
// Pure Interface — no data, no body
class IMeasurable {
public:
    virtual int getScore() const = 0;
    // "= 0" means: I declare this must
    // exist but I don't implement it.
    virtual ~IMeasurable() = default;
};
Interface Rule

ONLY pure virtual functions. No data members. No implementation. It is a pure contract — a list of promises that implementing classes must fulfill.

Lines 45–69 — Abstract ClassLAYER 2
class Assessment : public IMeasurable {
protected:
    string id, title;
    vector<Question> questions;
    int currentScore;          // ← Has real data
public:
    void addQuestion(...) { ... } // Real method

    virtual void conductAssessment() = 0;
    // ↑ Still abstract — subclasses must
    //   provide their own version

    int getScore() const override {
        return currentScore; // ← Implements IMeasurable
    }
};
✅ Why IMeasurable Was Created

Assessment and future classes (e.g., Assignment, Project) can all be stored in the same collection if they share the IMeasurable interface. Any code that needs a score just calls getScore() — it doesn't care what type of measurable thing it is.

💡 Real-World Analogy

A "Vehicle" interface says: all vehicles must have start() and stop(). A Car, Truck, and Motorcycle all implement it differently. You can command any vehicle to start without knowing how its engine works.

Inheritance

Child classes inherit properties and behaviour from parent classes — reuse without rewriting
📖 Definition

Inheritance is a mechanism where a new class (derived/child) acquires the properties and methods of an existing class (base/parent) using the : syntax. The child can use all inherited members and additionally define its own. It models "is-a" relationships.

Two separate inheritance trees in this file:

        cout << "TIME LIMIT: "

    }

Polymorphism

Same function name — different behaviour — correct version chosen automatically at runtime
📖 Definition

Polymorphism (Greek: "many forms") allows objects of different types to be treated as objects of a common base type. When a virtual function is called through a base class pointer, C++ automatically invokes the correct overridden version based on the actual runtime type of the object. This is called dynamic dispatch.

QuizzifyCore.cpp — Lines 172–195 — main()POLYMORPHISM
// A vector of BASE CLASS pointers — holds different derived types
vector<User*> systemUsers;         // ← stores User* not Student* or Admin*
systemUsers.push_back(&user1);    // user1 is a Student
systemUsers.push_back(&sysAdmin); // sysAdmin is an Admin

// One loop — calls SAME function — gets DIFFERENT results
for (const auto* user : systemUsers) {
    user->displayDashboard();        // Which version? Decided at RUNTIME!
}

// OUTPUT:
// ========================================
// 🎓 STUDENT DASHBOARD: Krish            ← Student's version ran
// Total EXP Score: 200
// ========================================
// ⚙️ SYSTEM ADMIN DASHBOARD: System Overseer ← Admin's version ran
virtual keyword enables this
class User {
public:
    virtual void displayDashboard() const {
        cout << "Welcome back, " << name;
    }
    // 'virtual' = enable dynamic dispatch
};

class Student : public User {
public:
    void displayDashboard() const override {
        cout << "🎓 STUDENT DASHBOARD";
        cout << "Score: " << totalScore;
    }
    // 'override' confirms we replace parent
};
❌ Without virtual — WRONG

If displayDashboard() is NOT virtual, calling it through a User* pointer would always run User::displayDashboard() — even if the object is a Student. This is called compile-time binding (wrong answer).

✅ With virtual — CORRECT

The vtable (virtual function table) is used at runtime. C++ looks at the actual object type, finds the right version, and calls it. Student gets Student's dashboard, Admin gets Admin's dashboard — automatically.

TypeWhen decidedMechanismExample in code
Runtime PolymorphismAt runtimevirtual + base pointeruser->displayDashboard()
Compile-time PolymorphismAt compile timeFunction overloadingNot used here but e.g. same function name, different parameters
💡 Real-World Analogy

You say "speak!" to a Dog and a Cat (both Animals). The Dog barks; the Cat meows. Same command — different reactions — based on the actual animal type. You don't need to know which animal it is before giving the command.

Smart Pointers — unique_ptr

Automatic memory management — no manual delete, zero memory leaks
📖 Definition

A smart pointer is a class that wraps a raw pointer and automatically manages the lifetime of the heap-allocated object. unique_ptr (from <memory>) ensures the object is deleted automatically when the pointer goes out of scope. Only one unique_ptr can own an object at a time.

Lines 180–184 — Smart Pointersunique_ptr
// ❌ OLD dangerous way:
Assessment* p = new PracticeQuiz("...");
// Must manually delete — easy to forget
delete p; // ← forget this = memory leak

// ✅ NEW safe way (used in this file):
unique_ptr<Assessment> practice =
    make_unique<PracticeQuiz>(
        "OS Fundamentals (Practice)"
    );
// ↑ Auto-deleted when function ends

practice->addQuestion(
    Question("What is a kernel?",
             {"Hardware","Core OS","App"},
             1)
);

unique_ptr<Assessment> exam =
    make_unique<CompetitiveTest>(
        "OS Fundamentals (Ranked)", 600
    );
✅ Why unique_ptr<Assessment> not unique_ptr<PracticeQuiz>?

The pointer type is the base class Assessment. This allows polymorphism — the same variable can hold a PracticeQuiz or a CompetitiveTest. If the type were unique_ptr<PracticeQuiz>, only PracticeQuiz could be stored.

→ Arrow Operator

practice->addQuestion() is shorthand for
(*practice).addQuestion(). It dereferences the pointer and accesses the member. Used with all pointers in C++.

🔑 Key Rule of unique_ptr

You cannot copy a unique_ptr — only move it. This enforces single ownership. That's why the function takes a const unique_ptr<Assessment>& (reference, not copy).

Lines 135–138 — Passing unique_ptr by reference
void recordAssessment(const unique_ptr<Assessment>& assessment) {
    assessment->conductAssessment(); // Calls correct version via polymorphism
    totalScore += assessment->getScore();
    quizzesTaken++;
}
💡 Real-World Analogy

Like a library book with an automatic return system. You borrow it (allocate), use it, and when you walk out of the library (scope ends), it automatically goes back to the shelf (memory freed). You never have to remember to return it yourself.

Virtual Destructor

Ensures proper cleanup of derived class objects when deleted through a base class pointer
📖 Definition

A virtual destructor ensures that when a derived class object is deleted through a base class pointer, the correct destructor chain executes — first the derived class destructor, then the base class destructor. Without it, only the base destructor runs, causing resource leaks.

Used in all 3 base classesDESTRUCTOR
class IMeasurable {
public:
    virtual ~IMeasurable() = default; // Line 16
};

class Assessment {
public:
    virtual ~Assessment() = default;  // Line 68
};

class User {
public:
    virtual ~User() = default;        // Line 124
};
Why it matters — example
User* u = new Student("U1","k@uni","K");

// WITHOUT virtual destructor:
delete u;
// → Only User::~User() runs
// → Student's variables NOT cleaned!
// → MEMORY LEAK 🔴

// WITH virtual destructor:
delete u;
// → Student::~Student() runs FIRST
// → Then User::~User() runs
// → Full cleanup ✅
= default — What does it mean?

= default tells the compiler: "generate the default implementation for this function". For destructors, the compiler writes an empty but correct destructor. It's cleaner than writing an empty ~ClassName() {} body manually.

Constructor Initializer List

Efficiently initialize members before the constructor body executes
📖 Definition

An initializer list appears after the constructor's parameter list, preceded by a colon (:). It directly initializes member variables before the constructor body {} runs. This is more efficient than assigning values inside the body, and is the only way to initialize const members, references, and call parent constructors.

Found throughout the file — key examplesINITIALIZER LIST
// Question constructor — Lines 30–31
Question(string q, vector<string> opts, int correctIndex)
    : text(q), options(opts), correctOptionIndex(correctIndex) {} // All 3 initialized

// Assessment constructor — Lines 53–55
Assessment(string t)
    : title(t), currentScore(0) {        // parent members initialized
    id = "ASSESS_" + to_string(time(0)); // id set in body (needs function call)
}

// CompetitiveTest — Lines 92 — calls BOTH parent + own initializer
CompetitiveTest(string t, int timeLimit)
    : Assessment(t),              // ← calls parent constructor
      timeLimitSeconds(timeLimit) {} // ← initializes own member

// Student — Line 133
Student(string id, string e, string n)
    : User(id, e, n),            // ← passes to parent
      totalScore(0), quizzesTaken(0) {}
✅ Why initializer list instead of assignment in body

When you assign inside the body, the member is first default-constructed, then overwritten — two operations. With an initializer list, the member is directly constructed with the value — one operation. For complex objects this makes a real performance difference.

Order of Initialization

Members are always initialized in the order they are declared in the class, not in the order of the initializer list. The parent constructor always runs before the child's members are initialized.

PDF Extraction — runs in browser, NOT serverKEY CONCEPT
const extractTextFromPdf = async (file: File) => {
  const pdfjsLib = await import('pdfjs-dist');        // Dynamic import
  pdfjsLib.GlobalWorkerOptions.workerSrc =
    `https://unpkg.com/pdfjs-dist@${pdfjsLib.version}/build/pdf.worker.min.js`;

  const arrayBuffer = await file.arrayBuffer();       // Read file as bytes
  const pdf = await pdfjsLib.getDocument({ data: new Uint8Array(arrayBuffer) }).promise;

  let fullText = "";
  for (let i = 1; i <= pdf.numPages; i++) {
    const page = await pdf.getPage(i);
    const textContent = await page.getTextContent();
    fullText += textContent.items.map((item: any) => item.str).join(" ") + "\n";
  }
  return fullText;   // Just text — much smaller than the PDF binary
};
⚠️ Why PDF is Parsed in the Browser (Critical Viva Question)

Vercel (the hosting platform) has a 4.5 MB body size limit for API requests. A PDF can be 10–70 MB. If we sent the raw PDF to the server, it would be rejected. By extracting just the text in the browser first, we only send a small string (~50 KB) to the API — well within limits.

dashboard/page.tsx — User Stats

Server-rendered page that queries the database and shows score, history, and weak topics
src/app/dashboard/page.tsxSERVER
export default async function DashboardPage() {
  const session = await getServerSession(authOptions);
  const userId = (session?.user as any)?.id;

  if (!userId) redirect("/login");    // Guard: not logged in → send to login

  // Query DB: get user + their last 5 quizzes
  const user = await prisma.user.findUnique({
    where: { id: userId },
    include: { quizzes: { orderBy: { createdAt: "desc" }, take: 5 } },
  });

  // Weak topics algorithm:
  const allTopicsRaw = user.quizzes.flatMap(q => JSON.parse(q.weakTopics || "[]"));
  // ["Photosynthesis", "Mitosis", "Photosynthesis"] → count each

  const topicCounts = allTopicsRaw.reduce((acc, t) => {
    acc[t] = (acc[t] || 0) + 1;   // {"Photosynthesis": 2, "Mitosis": 1}
    return acc;
  }, {});

  const sortedWeakTopics = Object.entries(topicCounts)
    .sort((a, b) => b[1] - a[1])   // Sort by frequency
    .slice(0, 5)                    // Top 5 only
    .map(entry => entry[0]);       // Just the topic name
}
Three stat cards shown:

1. Total Score — from user.totalScore in DB
2. Quizzes Taken — from user.quizzesTaken
3. Weak Topics — computed by the algorithm above — shows the 5 topics you get wrong most often

flatMap() Explained

flatMap() maps each quiz to its weak topics array, then flattens all those arrays into one big list.

Without flat: [["A","B"], ["A","C"]]
With flatMap: ["A","B","A","C"]
Then reduce counts each occurrence.

Why redirect: false?

Without it, signIn() does a full browser redirect. That means if there's an error (wrong password), you get sent to NextAuth's error page with no user-friendly message. With redirect: false, signIn returns a result object — we can read the error and show it in the UI ourselves.

API Routes — The Backend Logic

Next.js server functions that handle AI calls, database writes, and user registration
What is an API Route in Next.js?

A file named route.ts inside src/app/api/ becomes a server-side HTTP endpoint. When the frontend calls fetch('/api/generate'), this file's POST function runs on the server — it has access to environment variables, the database, and secret API keys that never reach the browser.

api/generate/route.ts — Gemini AI CallAI ROUTE
export async function POST(req: Request) {
  // 1. Auth guard — Gemini is locked behind login
  const session = await getServerSession(authOptions);
  if (!session) return NextResponse.json({ error: "Unauthorized" }, { status: 401 });

  // 2. Read the form data sent from the browser
  const formData = await req.formData();
  const extractedText = formData.get("text");          // PDF text
  const count = parseInt(formData.get("count") || "5");  // Question count
  const difficulty = formData.get("difficulty");       // easy/medium/hard/mixed

  // 3. Build and call Gemini API
  const genAI = new GoogleGenerativeAI(process.env.GEMINI_API_KEY);
  const model = genAI.getGenerativeModel({ model: "gemini-2.5-flash" });

  const prompt = `Generate exactly ${count} MCQs at ${difficulty} difficulty.
Output ONLY as raw JSON array: [{"question","options","answer","topic"}]
--- Source Text ---
${extractedText.slice(0, 30000)}`;  // Slice to stay within token limit

  const result = await model.generateContent({ contents: [{ role: "user", parts: [{ text: prompt }] }] });

  // 4. Clean any markdown wrappers Gemini might add
  const cleanedText = responseText.replace(/```json/g, "").replace(/```/g, "").trim();
  const questions = JSON.parse(cleanedText);  // Parse JSON string → JS array

  return NextResponse.json({ questions });
}
api/results/route.ts — Save to DatabaseDB ROUTE
export async function POST(req: Request) {
  const { score, total, mode, difficulty, weakTopics } = await req.json();
  const percentage = Math.round((score / total) * 100);

  // Create quiz record
  await prisma.quiz.create({
    data: { userId, score, total, percentage, mode, difficulty,
            weakTopics: JSON.stringify(weakTopics || []) }  // Store as JSON string
  });

  // Update user totals with WEIGHTED scoring formula:
  await prisma.user.update({
    data: {
      totalScore: { increment: score * 10 *
        (difficulty === 'hard' ? 2 : difficulty === 'medium' ? 1.5 : 1) },
      quizzesTaken: { increment: 1 }
    }
  });
  // Hard: score×10×2 | Medium: score×10×1.5 | Easy: score×10×1
}

Auth + Database — NextAuth & Prisma

How login sessions work and how data is stored persistently
src/lib/nextauth.ts — JWT Callback ChainAUTH CONFIG
// When Google user signs in: auto-create them in our DB
async signIn({ user, account }) {
  if (account?.provider === "google") {
    const existing = await prisma.user.findUnique({ where: { email: user.email } });
    if (!existing) await prisma.user.create({ data: { name, email, password: "" } });
  }
  return true;
},

// Step 2: Attach our DB user ID to the JWT token
async jwt({ token, user, account }) {
  if (user) token.id = user.id;    // First login: attach ID
  if (account?.provider === "google") {
    const dbUser = await prisma.user.findUnique({ where: { email: token.email } });
    if (dbUser) token.id = dbUser.id;  // Google: get our DB ID
  }
  return token;
},

// Step 3: Copy ID from token into session so API routes can use it
async session({ session, token }) {
  if (token) (session.user as any).id = token.id;  // Now available everywhere
  return session;
}
